# Device Side

## 1. 设备职责

设备负责三件事：

- 联网和绑定
- 下载配置和资源包
- 根据 `snapshot` 渲染宠物界面

设备不负责：

- 重建 CLI 原始事件
- 计算工作语义
- 维护账号和绑定关系真相

## 2. 设备输入

设备主要依赖三类输入：

- 本地保存的 Wi-Fi 和绑定信息
- Supabase 下发的配置与资源包信息
- MQTT 下发的精简 `snapshot`

`snapshot` 至少应包含：

- 当前宠物主状态
- 压缩后的 session 状态
- 焦点摘要
- 是否需要用户介入
- token 汇总

## 3. 启动状态

设备启动后建议进入以下状态之一：

- 未配网
- 配网中
- 联网中
- 未绑定
- 等待绑定确认
- 同步中
- 运行中
- 离线降级模式
- 错误状态

这样可以让界面和逻辑保持一致。

## 4. Wi-Fi 配网

推荐流程：

1. 长按按键进入配网模式
2. 设备开启热点，例如 `Buddy-XXXX`
3. 手机连接热点并打开本地配置页
4. 输入 Wi-Fi SSID 和密码
5. 设备保存配置并尝试联网
6. 联网成功后重启或自动进入联网状态
7. 界面显示联网成功

建议补充：

- 配网失败时自动回到配网模式
- 提供清除 Wi-Fi 入口
- 保存成功后先显示“正在连接”，不要直接静默重启

## 5. 设备绑定

推荐流程：

1. 设备联网后进入配对菜单
2. 设备用自己的长期 `device_id` 请求 Supabase
3. Supabase 返回一个短期 6 位绑定码
4. 绑定码进入“开放绑定”状态，并带过期时间
5. 用户在网页输入该绑定码完成绑定
6. 设备轮询或查询绑定结果
7. 绑定成功后，绑定码失效，设备显示绑定成功

建议规则：

- 不直接向用户暴露原始 MAC 地址
- 6 位绑定码应有过期时间
- 绑定码应有状态：`pending`、`bound`、`expired`
- 已绑定设备默认不重复开放绑定，除非用户主动重新配对

## 6. 联网后的同步流程

推荐顺序：

1. 读取本地 Wi-Fi
2. 联网
3. 读取本地绑定状态
4. 未绑定则提示去配对
5. 已绑定则请求 Supabase 获取设备配置
6. 检查当前资源包版本
7. 如有需要，下载或更新资源包
8. 连接 MQTT
9. 订阅自己的 `snapshot` topic
10. 进入主界面

## 7. 本地缓存

设备至少应保存：

- Wi-Fi 配置
- 绑定状态
- `device_id`
- 当前设备配置缓存
- 当前宠物资源包
- 宠物短期运行状态

缓存的意义是：

- 网络异常时仍可启动
- 资源下载失败时可继续使用旧资源
- 重启后可快速恢复

## 8. 资源包下载

资源包由设备自己下载，不经过 CLI。

建议规则：

- 设备根据 `asset_manifest_url` 和 `asset_version` 下载
- 下载完成后再切换到新资源
- 下载失败时继续使用旧缓存
- 如无缓存，使用内置默认资源

## 9. MQTT 行为

设备只订阅自己的 `snapshot` topic。

建议规则：

- 使用 retained `snapshot`
- 断线后自动重连
- 重连后自动重新订阅
- 收到最新 `snapshot` 后恢复当前界面

设备不应依赖历史事件流，只依赖当前快照。

## 10. 设置菜单

基础菜单建议包括：

- Brightness
- Volume
- Wi-Fi
- Pair
- Info
- Help
- Restart

设备信息建议显示：

- `device_id`
- Wi-Fi 状态
- MQTT 状态
- 固件版本
- 当前资源包版本

## 11. 异常回退

必须提前定义这些异常处理：

- 配网失败：回到配网模式
- 联网失败：显示离线状态并允许重试
- 绑定超时：绑定码失效并允许重新申请
- 配置拉取失败：使用本地缓存
- 资源包下载失败：继续使用旧缓存
- MQTT 连接失败：进入离线降级模式并自动重连

## 12. 当前边界

当前这份文档只定义设备系统逻辑，不展开宠物行为细节。

也就是说，这里只解决：

- 设备如何联网
- 设备如何绑定
- 设备如何同步配置和资源
- 设备如何接收并展示 `snapshot`

宠物自己的行为、情绪和动画策略应在另一份文档中单独定义。
