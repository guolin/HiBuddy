# Buddy 单宠物七状态动画规范

本文档定义设备端宠物动画的统一模型。  
V1 固定为单宠物、七状态、状态驱动的帧动画，不做多宠物切换，也不做大量离散表情库。

## 1. 总体原则

Buddy 的动画系统拆成两层：

- 资源层：定义某个动画状态有哪些帧
- 程序层：定义这些帧如何播放、多久切换、何时回退、何时重置

也就是说：

- 资源回答“画什么”
- 程序回答“怎么播”

V1 目标：

- 只有 1 个宠物角色
- 固定 7 个动画状态
- 每个状态有独立帧组
- 播放速率、循环、停顿、回退由程序控制
- 刷新逻辑参考 `claude-desktop-buddy`，优先低帧率局部刷新

## 2. 七个动画状态

V1 统一采用以下 7 个动画状态：

| 状态编号 | 状态名 | 用途 |
| --- | --- | --- |
| `S00` | `sleep` | 休眠、低活跃、呼吸感 |
| `S01` | `idle` | 正常待机、轻微观察 |
| `S02` | `busy` | 正在工作、持续活跃 |
| `S03` | `attention` | 需要用户注意、提醒 |
| `S04` | `celebrate` | 完成、开心、庆祝 |
| `S05` | `dizzy` | 错误、异常、离线降级 |
| `S06` | `heart` | 短时喜爱/奖励特效 |

说明：

- `heart` 是短时特效状态，不是常驻状态
- `sleep` 和 `idle` 必须视觉上明显不同
- `busy` 和 `attention` 也必须节奏不同
- `dizzy` 用于错误或离线降级，但不能做得过于惊悚

## 3. 业务状态到动画状态映射

设备运行态先产生业务状态，再映射到动画状态。

推荐映射：

| 业务状态 | 动画状态 |
| --- | --- |
| `Idle` | `idle` |
| `Thinking` | `busy`，必要时可退到 `idle` |
| `Busy` | `busy` |
| `Waiting` | `attention` |
| `requiresUser=true` | `attention` |
| `Done` | `celebrate` |
| `Error` | `dizzy` |
| `OfflineDegraded` | `dizzy` |
| `ScreenSleep` | `sleep` |

补充规则：

- `heart` 只能由短时事件触发，例如奖励、配对成功、欢迎动画
- `heart` 播完后必须自动回退到基础状态
- 当前基础状态由业务映射得出，不被 `heart` 永久覆盖

## 4. 资源编号规则

V1 不做“全局连续编号”，统一采用两级编号：

- 状态编号：`Sxx`
- 帧编号：`Sxx_Fyy`

示例：

- `S00_F00`：`sleep` 的第 1 帧
- `S00_F01`：`sleep` 的第 2 帧
- `S01_F00`：`idle` 的第 1 帧
- `S04_F03`：`celebrate` 的第 4 帧

推荐目录概念：

```text
pet-pack/
  manifest.json
  states/
    S00_sleep/
    S01_idle/
    S02_busy/
    S03_attention/
    S04_celebrate/
    S05_dizzy/
    S06_heart/
```

V1 文档阶段不强制具体文件格式，但编号规则必须固定不变。

## 5. 每个状态的资源建议

### 5.1 `S00 sleep`

- 建议帧数：`3 ~ 4`
- 播放方式：循环，可往返
- 节奏目标：缓慢呼吸、低扰动
- 建议内容：闭眼、轻微起伏、偶发慢呼吸

### 5.2 `S01 idle`

- 建议帧数：`4 ~ 6`
- 播放方式：循环，带轻微停顿
- 节奏目标：待机但活着，不僵硬
- 建议内容：眨眼、轻微左右看、细小 bob

### 5.3 `S02 busy`

- 建议帧数：`4 ~ 6`
- 播放方式：循环
- 节奏目标：持续工作、更积极
- 建议内容：眼神集中、动作频率更高、姿态更紧凑

### 5.4 `S03 attention`

- 建议帧数：`3 ~ 5`
- 播放方式：循环
- 节奏目标：明确提醒用户
- 建议内容：抬头、张望、明显提示姿势

### 5.5 `S04 celebrate`

- 建议帧数：`4 ~ 6`
- 播放方式：短循环，允许播 1~2 轮
- 节奏目标：开心、完成感
- 建议内容：跳动、笑眼、兴奋动作

### 5.6 `S05 dizzy`

- 建议帧数：`3 ~ 5`
- 播放方式：循环
- 节奏目标：异常但不过度夸张
- 建议内容：晕、低落、失去连接感

### 5.7 `S06 heart`

- 建议帧数：`3 ~ 4`
- 播放方式：非永久循环，只播 1~2 轮
- 节奏目标：奖励、喜欢、欢迎
- 建议内容：爱心、拥抱感、闪亮眼神

## 6. 动画播放器程序规则

程序层必须显式管理状态机，不再使用“每个业务状态只切两帧”的简化模型。

建议播放器配置项：

- `frameIntervalMs`
- `holdOnLastFrameMs`
- `loopMode`
- `playCount`
- `randomIdleVariant`

建议 `loopMode`：

- `loop`
- `ping_pong`
- `play_once`
- `play_n_then_fallback`

建议行为：

1. 状态切换时，动画序列从第 1 帧重置
2. 如果是短时状态，例如 `heart`，播完后自动回到当前基础状态
3. `idle` 可以允许少量随机变体，例如随机眨眼或额外停顿
4. 状态切换优先级必须清晰：
   - 短时特效结束后回退
   - 新业务状态到来时可立即打断旧状态

## 7. 推荐默认播放节奏

推荐默认节奏如下：

| 状态 | 推荐帧间隔 |
| --- | --- |
| `sleep` | `350 ~ 500ms` |
| `idle` | `220 ~ 320ms` |
| `busy` | `120 ~ 180ms` |
| `attention` | `140 ~ 220ms` |
| `celebrate` | `100 ~ 160ms` |
| `dizzy` | `160 ~ 240ms` |
| `heart` | `120 ~ 180ms` |

额外规则：

- `sleep` 应最慢
- `busy` 应明显快于 `idle`
- `attention` 比 `busy` 稍慢，但更有“强调感”
- `heart` 不长期持续

## 8. 刷新逻辑

刷新逻辑参考 `claude-desktop-buddy`，采用“低帧率宠物更新 + 局部刷新”的思路。

V1 建议：

- 宠物区域默认刷新基线约 `5 FPS`
- 即普通动画步进按约 `200ms` 一次
- 状态变更时立即刷新宠物区域
- 页面切换、菜单弹出、guide 切换时允许整页刷新
- 平稳运行时不做“整屏每帧重画”

渲染策略：

- 宠物区、文本区、状态灯区逻辑分离
- 普通动画只更新宠物区域
- 文本变化时只更新相关文本区域
- 整体保持低频、稳定、可预测

## 9. 资源制作建议

V1 推荐仍然保持轻量：

- 原始帧建议：`64x64`
- 显示尺寸：放大到 `128x128`
- 放大方式：nearest-neighbor
- 优先像素风或低色深风格

理由：

- 资源更轻
- 制作成本更低
- 后续替换和版本管理更简单

## 10. 实现验收标准

后续实现应满足以下标准：

- 七状态名称和编号固定不变
- 每次状态切换都从该状态首帧开始播放
- `heart` 播完后自动回退
- `sleep` 与 `busy` 的视觉节奏明显不同
- 常规动画按约 `200ms` 节奏更新
- 普通动画不依赖整屏每帧重绘
- 菜单、主页面、guide 切换不会破坏宠物动画状态机

## 11. V1 不做的事情

V1 不做以下能力：

- 多宠物切换
- 40+ 独立表情库
- 实时 GIF 解码作为主方案
- 骨骼动画
- AI 实时生成逐帧图片
- 复杂粒子系统

V1 先把“单宠物七状态动画播放器”做稳。
