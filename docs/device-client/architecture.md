# Buddy Device Client Architecture

本文档定义设备客户端的基础架构、模块拆分、边界和运行顺序。

目标是回答三个问题：

- 设备端到底负责什么
- 设备端内部该如何拆模块
- 设备端启动后按什么顺序运行

## 1. 产品定位

Buddy 设备客户端是运行在 `M5StickS3` 上的轻量 AI 宠物系统。

它不是：

- 审批终端
- 聊天终端
- 长文本浏览器
- 上游事件分析器

它是：

- 状态展示器
- 宠物动画播放器
- 提醒设备
- 本地交互和联网终端

V1 默认不是：

- 审批执行终端
- 设备侧直接决策终端

## 2. 设备端职责

设备端负责以下事情：

- 启动和自检
- 屏幕渲染
- 本地按键交互
- Wi-Fi 配网
- 设备绑定
- 下载宠物资源包
- 订阅 MQTT `snapshot`
- 根据当前状态驱动宠物动画
- 维护本地短期演出状态
- 本地缓存与离线回退

设备端不负责：

- 重建原始 CLI 事件
- 推断复杂业务真相
- 在 V1 中直接执行审批用户操作
- 维护账号系统
- 生成上游工作摘要

## 3. 外部输入

设备端主要有四类输入：

### 3.1 本地输入

- `A` 键短按和长按
- `B` 键短按和长按
- IMU 动作输入
- 唤醒和休眠

### 3.2 本地持久化数据

- Wi-Fi 配置
- `device_id`
- `device_token`
- 当前设备配置缓存
- 当前宠物资源包
- 当前宠物短期状态
- 最近一次 `snapshot`

### 3.3 控制面输入

来自 Supabase Edge Functions：

- 配对码
- 绑定状态
- MQTT 连接信息
- 显示配置
- 激活宠物资源信息

### 3.4 消息面输入

来自 MQTT 的精简 `snapshot`：

- `pet_state`
- `session_states`
- `focus_title`
- `requires_user`
- `total_tokens`
- `today_tokens`

V2 如需设备直接审批，可额外引入：

- `pending_action`
- `prompt_id`
- `action kind`
- `action result`

## 4. 输出

设备端主要输出以下结果：

- 屏幕页面
- 宠物动画
- 绑定码和帮助信息
- 心跳与宠物版本同步
- 本地缓存更新

V1 暂不要求设备端输出审批或复杂控制指令。

## 5. 模块拆分

建议设备端按以下模块组织：

### 5.1 System

负责：

- 启动流程
- 时间片调度
- 电源与亮度
- 休眠与唤醒
- 全局状态机

### 5.2 Input

负责：

- `A/B` 键扫描
- 长按和短按识别
- IMU 姿态和摇晃识别
- 首次唤醒拦截

### 5.3 UI Router

负责：

- 主界面切换
- 菜单导航
- 子页面进入和返回
- 临时提醒页覆盖

### 5.4 Pet Runtime

负责：

- 载入资源包
- 当前状态帧循环
- 本地微动作
- 局部刷新
- 程序特效

### 5.5 Snapshot Runtime

负责：

- 接收最新 `snapshot`
- 状态映射
- 生成宠物主状态
- 维护提醒触发器

V1 的 `waiting` 只负责提醒，不负责在设备上展示完整审批内容。

### 5.6 Network

负责：

- Wi-Fi 连接
- HTTP API 调用
- MQTT 连接与重连

V2 如引入设备侧审批，可在该模块新增“发送审批动作消息”的能力。

### 5.7 Storage

负责：

- 本地配置读写
- 资源包下载缓存
- 安全切换新资源

## 6. 全局状态机

建议设备运行时至少存在以下全局状态：

- `BOOT`
- `NO_WIFI`
- `WIFI_PORTAL`
- `CONNECTING_WIFI`
- `NO_BINDING`
- `PAIRING`
- `SYNCING`
- `RUNNING`
- `OFFLINE_DEGRADED`
- `ERROR`
- `SCREEN_SLEEP`

这些状态是系统级状态，不等于宠物状态。

如果后续加入 V2 审批动作，可新增：

- `ACTION_PENDING`
- `ACTION_SENDING`
- `ACTION_RESULT`

## 7. 启动顺序

推荐启动顺序如下：

1. 初始化硬件
2. 读取本地配置
3. 初始化屏幕和默认 UI
4. 检查 Wi-Fi 是否已配置
5. 若无 Wi-Fi，则进入配网流程
6. 若有 Wi-Fi，则尝试联网
7. 联网成功后检查是否已有绑定信息
8. 未绑定则进入配对流程
9. 已绑定则调用 `device-bootstrap`
10. 拉取配置与激活宠物信息
11. 检查本地宠物版本
12. 必要时下载资源包
13. 连接 MQTT
14. 载入上次缓存的 `snapshot`
15. 进入主界面运行

## 8. 运行时优先级

渲染顺序建议遵循以下优先级：

1. 系统错误
2. 配网/配对关键步骤
3. 临时提醒
4. 菜单与设置
5. 主界面常驻显示

也就是说，当设备处于关键绑定流程或错误状态时，不应继续优先展示普通宠物待机动画。

## 9. 页面层与宠物层分离

设备端要明确拆分：

- 页面层
- 宠物层

页面层负责：

- 页面框架
- 文案
- 选中态
- 网络状态展示

宠物层负责：

- Sprite 帧动画
- 程序位移
- 表情切换
- 提醒态特效

不要把 UI 逻辑写进宠物动画模块里。

## 10. 推荐主循环结构

推荐按固定节奏轮询，不需要复杂 RTOS 编排。

建议拆成多个节拍：

- `input tick`: `16ms`
- `animation tick`: `66ms` 到 `100ms`
- `ui tick`: `33ms`
- `network poll tick`: `250ms` 到 `1000ms`
- `heartbeat tick`: `60s` 到 `180s`
- `pet sync tick`: `600s`

## 11. V1 默认能力范围

V1 必须实现：

- 三个主界面
- 配网
- 配对
- 本地菜单
- 宠物资源加载
- MQTT snapshot 驱动
- 断网降级
- IMU 的轻交互能力

V1 可以后补：

- 更复杂的宠物行为参数
- 多宠物切换
- 商城和资源购买
- 声音主题包
- 设备端语音
- 设备侧审批执行
